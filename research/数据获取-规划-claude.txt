很好的切入点，数据获取模块是整个系统的地基，设计好了后面省很多麻烦。先直接测一下这三个页面的结构。容器内网络不通，符合预期。基于 GitHub Trending 的实际页面结构直接给你完整分析。先看数据库 ER 图，再逐一分析每个问题。---

## 表结构设计说明

**6张核心表，职责分明：**

`repositories` 是主实体表，存每个 repo 的静态属性。`total_stars` 是当前全量值，不是增量，便于随时跟踪绝对体量。`first_seen_at` 记录首次进热榜时间，对分析"新兴项目 vs 常驻项目"很有价值。

`trending_snapshots` 是核心时序表，每次爬取都往这里插一条记录，记录该 repo 在某个窗口类型（daily/weekly/monthly）、某个日期的排名和 stars 增量。这是整个系统最重要的历史数据，绝对不能做 upsert 覆盖，必须每次 insert 保留快照。`window_type + snapshot_date + repo_id` 建联合唯一索引。

`crawl_jobs` 记录每次爬取任务的执行情况，包括状态、发现了多少 repo、有多少是新增的、有没有报错。这张表是运维利器，管理平台里的"任务日志"页面直接读它。

`articles` 存 LLM 生成的文章，`content_md` 是原始 markdown，`content_html` 是渲染后用于推送的 HTML，`wechat_media_id` 是上传到微信素材库后返回的 ID（群发时要用这个）。`status` 枚举：`draft / reviewing / published / failed`。

`article_repos` 是 article 和 repository 的多对多关联表，记录每篇文章选了哪些 repo、排名第几。便于后续分析"哪个 repo 被推过几次"。

`push_logs` 记录每次推送的完整请求/响应，方便排查微信 API 报错。

## 爬取频率设置

**核心原则：爬取频率要比推送频率高，但没必要实时。**

推荐策略如下：

daily 窗口每天 **07:15 爬取一次**，给出文留 45 分钟缓冲，08:00 准时推送。不需要多次爬取，GitHub Trending daily 本来就是每天刷新一次，早上爬到的就是最新状态。

weekly 窗口每天 **07:20 也爬一次**（和 daily 同时跑），虽然周报只在周一推，但每天的快照数据都入库——这样你能看到整周内同一批 repo 的排名变化，管理平台里能画出"某 repo 本周排名曲线"，数据更丰富。

monthly 窗口同理，每天 **07:25 爬一次**，月报在每月1号推，但31天的快照都保留着。

```
# crontab 设置（服务器时间比推送时间早45分钟）
15 7 * * *  python crawl.py --window daily
20 7 * * *  python crawl.py --window weekly  
25 7 * * *  python crawl.py --window monthly
```

这样每天总共爬三次，每次一个页面，对 GitHub 服务器几乎没有压力，被封的概率极低。

## 爬取难易程度

**整体来说：中等偏简单，但有几个坑要提前知道。**

GitHub Trending 页面是**服务端渲染的静态 HTML**，不是 SPA，不需要 Selenium/Playwright 去执行 JavaScript，直接用 `requests` + `BeautifulSoup` 就够。这是最省事的爬取场景。

页面结构非常稳定，核心选择器是：

```python
# 每个 repo 是一个 <article class="Box-row"> 标签
articles = soup.find_all('article', class_='Box-row')

for art in articles:
    # repo 全名（owner/name）
    repo_full = art.find('h2').find('a')['href'].strip('/')
    
    # 描述
    desc_el = art.find('p', class_='col-9')
    description = desc_el.text.strip() if desc_el else ''
    
    # 主语言
    lang_el = art.find(itemprop='programmingLanguage')
    language = lang_el.text.strip() if lang_el else ''
    
    # 今日/本周 stars 增量
    # 文本格式：" 1,234 stars today" 或 " 567 stars this week"
    stars_el = art.find('span', class_='d-inline-block float-sm-right')
    stars_gained = parse_stars(stars_el.text) if stars_el else 0
    
    # 总 stars（需要去掉逗号）
    total_stars_el = art.find('a', href=lambda h: h and 'stargazers' in h)
    total_stars = int(total_stars_el.text.strip().replace(',', '')) if total_stars_el else 0
```

**已知的坑：**

GitHub 会对高频请求返回 429 或者直接返回空页面（但你一天只请求3次，完全不会触发）。偶尔 HTML 结构会因为 GitHub 改版而变，建议把 `raw_html_snippet` 存进数据库，改版时方便 debug，不用重新爬历史数据。另外 stars 文本里的数字要处理 `k` 单位，比如 `"1.2k stars today"` 需要转换成 `1200`。

## 网络因素

这是这个项目**最实质的风险**，需要认真对待。

**从国内云服务器访问 github.com 会不稳定。** 具体表现是：有时候正常，有时候超时，有时候返回不完整的 HTML。如果你的云服务器在国内（阿里云、腾讯云、华为云的国内节点），建议以下几个方案选一个：

一是**用海外节点的云服务器**（推荐）。新加坡或香港节点，访问 GitHub 稳定，延迟 30-80ms，爬虫完全没问题。阿里云轻量应用服务器新加坡节点一个月约 24-40 元，比折腾代理省心太多。

二是如果必须用国内服务器，加一个**代理池或 HTTP 代理中转**。在爬虫里设置 `proxies` 参数，走代理出海，但代理本身有维护成本。

三是**备用方案**：GitHub 有个非官方但被广泛使用的镜像 API——`https://api.github.com/search/repositories?q=...&sort=stars&order=desc`，虽然搜索接口不能完全替代 Trending 页面（Trending 的算法是"增速"不是"绝对数"），但可以作为爬取失败时的降级数据源。

代码里一定要加重试逻辑和失败告警：

```python
import time, requests
from requests.adapters import HTTPAdapter
from urllib3.util.retry import Retry

def get_session():
    s = requests.Session()
    retry = Retry(total=3, backoff_factor=2, status_forcelist=[429, 500, 502, 503])
    s.mount('https://', HTTPAdapter(max_retries=retry))
    s.headers.update({
        'User-Agent': 'Mozilla/5.0 (compatible; personal-bot/1.0)',
        'Accept-Language': 'en-US,en;q=0.9',
        'Accept': 'text/html,application/xhtml+xml'
    })
    return s

def crawl_with_fallback(window: str):
    url = f"https://github.com/trending?since={window}"
    try:
        resp = get_session().get(url, timeout=30)
        resp.raise_for_status()
        return parse_html(resp.text)
    except Exception as e:
        # 写入 crawl_jobs 失败状态，触发告警（微信/邮件）
        log_crawl_failure(window, str(e))
        return []
```

爬取失败时不要让整个任务崩掉，只记录失败状态，等第二天重试，或者在管理平台里手动触发补爬。
